# ADR-0002: 将来の管理画面は同一 Workers アプリ内に置き Cloudflare Access で保護する

## Context

既存の公開サイトは、Cloudflare Workers 上で R2 の Markdown とアセットを配信する構成です。

次フェーズでは、公開ブログ運用を続けたまま、記事の作成、編集、削除、プレビューをブラウザ上から扱えるようにする要求が既に整理されています。

このとき、管理機能を別サービスとして切り出すか、同じアプリの中に同居させるか、認証責務をどこまで Access に寄せるかが将来の構成判断になります。

## Decision

将来の管理画面は、既存の公開サイトと同じ Cloudflare Workers アプリの `/admin/*` 配下に実装します。

認証の第一防衛線は Cloudflare Access とし、必要な場合のみ Workers 側で `cf-access-jwt-assertion` を追加検証します。

記事の CRUD とプレビューは、既存の `POSTS_BUCKET` を直接利用する前提で進めます。

## Rationale

同一アプリ内に管理機能を置けば、公開表示と管理画面で Markdown レンダラーや R2 アクセス契約を共有しやすくなります。

Cloudflare Access を前段に置くことで、個人運用サイトに対して過剰な認証基盤を追加せずに保護境界を設けられます。

また、記事ソースを既存の R2 に残すことで、公開配信と編集運用のストレージを分断せずに済みます。

## Alternatives Considered

### 管理画面を別サービスまたは別リポジトリとして切り出す

公開系と管理系を分離しやすい利点はありますが、個人サイトとしてはデプロイ面と保守面のコストが増えます。

Markdown レンダラーや R2 契約の二重管理も起こりやすいため採用しませんでした。

### 管理画面用にデータベースまたは外部 CMS を導入する

一般的な管理 UI は作りやすくなりますが、現行プロジェクトの R2 ベース運用から外れます。

公開配信基盤と執筆基盤の距離が広がり、移行コストに対して現時点の便益が小さいため採用しませんでした。

## Consequences

`/admin/*` の認証境界、R2 への書き込み権限、公開レンダラーとの共有責務を明確に保つ必要があります。

Cloudflare Access だけで十分か、Workers 側 JWT 検証を常に有効にするかは、導入時の運用条件に応じて詰める必要があります。

Markdown 拡張やプレビュー機能は、公開ページと同一の変換パイプラインを中心に設計する前提になります。
