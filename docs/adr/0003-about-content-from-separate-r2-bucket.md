# ADR-0003: About 詳細ページは `RESUME_ASSETS_BUCKET` の特定 Markdown を表示する

## Context

既存の公開サイトでは、記事本文と画像アセットを `POSTS_BUCKET` から読み出し、`/posts` と関連 API で配信しています。

issue #12 では、トップページ内の簡易な自己紹介だけでなく、`/about` にスキル棚卸しやレジュメ相当の情報をまとめた詳細ページを追加する必要があります。

このとき、about 用コンテンツを既存の `POSTS_BUCKET` に混在させるか、リポジトリに静的ファイルとして持つか、別ストレージに分けるかを決める必要がありました。

また、issue #12 を chat や一時的な plan に依存せず実装できるようにするため、初版 `/about` がどのバケットのどのオブジェクトを読むのか、失敗時にどう振る舞うのかまで固定する必要がありました。

## Decision

初版 `/about` は、R2 binding 名 `RESUME_ASSETS_BUCKET` から Markdown を取得します。

`RESUME_ASSETS_BUCKET` は Cloudflare R2 バケット `portfolio-resume-assets` にバインドします。

初版 `/about` が取得するオブジェクトキーは `resume/skills_20250728161256.md` の 1 件に固定します。

`/about` は取得した Markdown を既存の Markdown 変換基盤で HTML に変換して表示します。

このオブジェクトの取得または Markdown 変換に失敗した場合、`/about` は部分表示を行わずエラー画面を返します。

初版では、about 用オブジェクトの動的列挙、複数 Markdown の組み立て、表示順の設定 UI は扱いません。

## Rationale

プロフィール資料は記事一覧や記事詳細とは公開契約も更新単位も異なるため、記事用バケットと分離した方が責務が明確になります。

初版の取得対象を `resume/skills_20250728161256.md` に固定すれば、実装者は chat を見ずに `wrangler.jsonc`、型定義、ローダー、テストを具体化できます。

また、初版で 1 ファイルに限定することで、複数ドキュメントの優先順位や必須/任意判定を未決のまま持ち込まずに済みます。

## Alternatives Considered

### about 用 Markdown を `POSTS_BUCKET` に混在させる

既存の R2 バインディングをそのまま使える利点はありますが、公開記事のコンテンツ契約とプロフィール資料の契約が同じ場所に混在します。

記事一覧・サイトマップ・タグ導線とは無関係なファイルを同一バケットの命名規則で管理することになり、将来の運用境界が曖昧になるため採用しませんでした。

### about 内容をリポジトリ内の静的ファイルとして持つ

実装は簡単ですが、R2 ベースで外部コンテンツを運用する現在の更新フローから外れます。

プロフィール資料だけ更新経路が別になると、記事と同じくストレージ上の Markdown を参照する運用方針から外れ、一貫性が下がるため採用しませんでした。

### 初版から複数 Markdown を列挙して組み立てる

将来的には必要になるかもしれませんが、今回の入力として確定しているのは `resume/skills_20250728161256.md` の 1 件だけです。

この段階で列挙ルールや複数ファイル契約まで拡張すると、未確定事項を docs に持ち込むことになるため採用しませんでした。

## Consequences

Workers の設定と型定義に、`RESUME_ASSETS_BUCKET` を追加する必要があります。

`/about` は `RESUME_ASSETS_BUCKET` から `resume/skills_20250728161256.md` を取得する専用ローダーを持ちますが、Markdown パーサー自体は既存の変換基盤を再利用できます。

初版では、対象オブジェクト、失敗時方針、非対象範囲が docs に固定されたため、実装者は追加質問なしで route、loader、tests を起こせます。
