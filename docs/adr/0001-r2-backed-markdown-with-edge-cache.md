# ADR-0001: R2 上の Markdown を Workers で配信し Cache API で前段キャッシュする

## Context

このリポジトリは、Poko Hanada のポートフォリオと公開ブログを 1 つの Cloudflare Workers アプリで配信しています。

記事本文と画像アセットは R2 バケット `POSTS_BUCKET` に保存され、公開時には一覧表示、記事詳細表示、画像配信、サイトマップ生成から参照されます。

単純な静的サイト生成だけでは記事追加時の柔軟性が下がり、一方でデータベースや CMS を導入すると個人サイトとしては運用負荷が大きくなります。

## Decision

記事本文は Markdown ファイル、画像などの関連アセットは R2 オブジェクトとして保持し、HonoX/Hono でリクエスト時に読み出して配信します。

公開アクセス時の R2 読み出しは Cloudflare Cache API を前段に置き、記事本文とアセットの両方をキャッシュします。

## Rationale

Markdown + R2 であれば、記事本文をテキストとして扱いやすく、差分管理やローカル同期も容易です。

Cloudflare Workers 上でそのまま配信すれば、別の API サーバーやデータベースを持たずに公開サイトを完結できます。

さらに Cache API を併用することで、同じ記事やアセットへの再アクセス時に R2 への往復を減らし、静的配信に近い体感を得られます。

## Alternatives Considered

### リポジトリ内コンテンツをビルド時に静的生成する

小規模サイトでは有効ですが、記事追加のたびに再ビルドと再デプロイが必要になります。

現行リポジトリには R2 シード運用が既に存在しており、コンテンツストアを別へ戻す合理性が薄いため採用しませんでした。

### データベースまたはヘッドレス CMS を導入する

柔軟な管理 UI を作りやすくなりますが、現行フェーズで必要な公開機能に対して構成が重くなります。

個人運用のポートフォリオとしては、インフラ要素と保守対象が増えすぎるため採用しませんでした。

## Consequences

記事の公開契約は R2 上の Markdown と frontmatter に依存するため、そのフォーマット互換性を保つ必要があります。

公開性能は Cache API の恩恵を受けますが、キャッシュキー設計やキャッシュ無効化の考慮が必要になります。

将来、管理画面や複雑なワークフローが必要になった場合は、R2 中心の運用を維持するか別の永続化手段へ移るかを再判断する必要があります。
